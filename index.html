BFS(breadth-first search)

  • Introduction
    • Breadth-first search (BFS) is an algorithm used for tree traversal on graphs or tree data structures. BFS can be easily implemented using recursion and data structures like dictionaries and lists.
  • Implement
    • Pick any node, visit the adjacent unvisited vertex, mark it as visited, display it, and insert it in a queue.
    • If there are no remaining adjacent vertices left, remove the first vertex from the queue.
    • Repeat step 1 and step 2 until the queue is empty or the desired node is found.

DFS(depth-first search)

  • Introduction
    • The Depth-First Search is a recursive algorithm that uses the concept of backtracking.
  • Implementation
    • We will start by putting any one of the graph’s vertex on top of the stack.
    • After that take the top item of the stack and add it to the visited list of the vertex.
    • Next, create a list of that adjacent node of the vertex. Add the ones which aren’t in the visited list of vertexes to the top of the stack.
    • Lastly, keep repeating steps 2 and 3 until the stack is empty.

Dijkstra’s algorithm

  • Introduction
    • Given a graph and a source vertex in the graph, find the shortest paths from the source to all vertices in the given graph.
  • Implementation
    • Create a set sptSet (shortest path tree set) that keeps track of vertices included in the shortest-path tree, i.e., whose minimum distance from the source is calculated and finalized. Initially, this set is empty.
    • Assign a distance value to all vertices in the input graph. Initialize all distance values as INFINITE. Assign distance value as 0 for the source vertex so that it is picked first.
    • While sptSet doesn’t include all vertices
      • Pick a vertex u which is not there in sptSet and has a minimum distance value.
      • Include u to sptSet.
      • Update distance value of all adjacent vertices of u. To update the distance values, iterate through all adjacent vertices.

面向对象和面向过程的区别

面向过程

  • 面向过程是一种以事件为中心的编程思想，编程的时候把解决问题的步骤都分析出来，然后用函数将其解决
  • 例子
    • 五子棋 如下，一步步调用函数:
        1. 开始游戏
        1. 黑子先走
        1. 绘制图面
        1. 判断输赢
        1. 轮到白子
        1. 绘制图面
        1. 判断输赢
        1. 返回步骤2
  • 优点:
    • 流程化使得编程任务明确
    • 效率高
  • 缺点:
    • 需要深入思考
    • 代码复用性差
    • 后期维护难度大

面向对象

  • 将每个人和事物都看做对象，每个对象有自己的属性和行为。
  • 例子
    • 五子棋:
        1. 黑白双方
        1. 棋盘系统
        1. 规则系统
  • 优点:
    • 结构清晰, 程序是模块化和结构化，更加符合人类的思维方式；
    • 易拓展, 代码重用率高，可继承，可覆盖，可以设计出低耦合的系统
    • 易维护, 系统低耦合的特点有利于减少程序的后期维护工作量
  • 缺点:
    • 开销大，当要修改对象内部时，对象的属性不允许外部直接存取，所以要增加许多没有其他意义、只负责读或写的行为.
    • 性能低，由于面向更高的逻辑抽象层，使得面向对象在实现的时候，不得不做出性能上面的牺牲，计算时间和空间存储大小都开销很大

java的内存回收机制

  • 为什么要进行内存回收?
    • C、C++等语言中，内存的分配和释放由程序代码来完成，容易出现由于程序员漏写内存释放代码引起的内存泄露，最终导致系统内存耗尽。
    • 内存泄漏:
      • 内存泄漏（Memory Leak）是指程序中已动态分配的堆内存由于某种原因程序未释放或无法释放，造成系统内存的浪费，导致程序运行速度减慢甚至系统崩溃等严重后果。

内存标记方式

  • 引用计数法
    • 当对象被创建并赋值给变量的时候计数为1; 当有其他变量引用该对象的时候, 引用计数+1; 当引用对象的变量超出存活范围或释放对象引用的时候, 引用计数-1。对象引用计数为0的时候代表可以被垃圾收集器回收。
  • 根搜索算法
    • 从一些指定的根对象出发，遍历找到和这个根对象有引用关系的对象, 从这些对象开始继续寻找，从而形成一个个的引用链, 不在引用链上的对象标记成为不可达的对象, 也就是需要回收的对象。

内存回收方式

  • 标记-清理算法(mark and sweep)
    • 方法: 采用跟踪算法标记内存对象后, 扫描堆内存中未被标记的对象, 进行回收。算法不移动对象, 对不存活对象进行回收。
    • 造成的问题: 会引起内存碎片
  • 标记-整理算法(compacting)
    • 方法: 和标记-清理方法相同, 但是会把存活的对象向内存前半部分空闲区域移动, 并更新对象的指针
    • 造成的问题: 执行成本较高, 但是可以解决内存碎片的问题
  • 复制-清除算法(copying)
    • 该算法把内存分为空闲区和对象区，新建对象存储到对象区中。当对象区满时，先采用跟踪算法对对象进行标记，再把存活对象拷贝到空闲区，清空原对象区，空闲区和对象区互换角色。在拷贝过车中，程序需要暂停，此算法适用于存活对象叫少的情况，可以解决内存碎片问题。
    • 造成的问题: 把内存分为两部分, 会有一部分没法使用
  • 分代回收算法
    • 会根据对象的存活周期将内存区域分为三大块: 年轻代(Eden区, S0, S1), 老年代, 永久代
      1. 对象首次创建进行内存分配的时候，首先会放置在 Eden 区，当 Eden 区放满了或者当该对象太大无法放进 Eden 区的时候，此时会对年轻代（Eden区 和 S0）进行一次 GC，将幸存下来的对象放置在 S1，然后清空掉 Eden区和 S0 区；（此时年轻代采用的是 复制算法）
      1. 在上面第一步中对年轻代进行垃圾回收的时候，同时会对幸存的对象进行标记，统计每个幸存对象经历的 GC 次数；
      1. 当 S1 区满了之后，或者年轻代的对象经历过指定次数的 GC 之后，这部分对象会被放置到老年代之中；
      1. 当老年代也满了之后，便会对老年代进行一次 GC；（老年代采用的是 标记-整理算法）

垃圾回收器

  • 串行垃圾回收器（Serial Garbage Collector）；
    • 串行垃圾回收器会暂停所有的应用程序线程，并采用单独的的线程进行 GC。
    • 适用于单 CPU、并且对应用程序的暂停时间要求不高的情况，所以不太适合当前的生产环境。
  • 并行垃圾回收器（Parallel Garbage Collector）；
    • 并行垃圾回收器是 JVM 默认的垃圾回收器，相较于串行垃圾回收器而言性能稍有提升，它也是需要暂停所有的应用程序线程的，但是区别是它会使用多线程进行 GC。
    • 所以并行垃圾回收器适用于多 CPU 的服务器、并且能接受短暂的应用暂停的程序
  • 并发标记扫描垃圾回收器（CMS Garbage Collector）；
    • CMS 回收器也是一种并行的垃圾回收器，它会采用多线程来进行扫描堆内存，标记需要清理的对象并将这些对象清理掉。
    • 但是 CMS 它需要更多的 CPU 来保证程序的吞吐量，并且它保证了最短的回收停顿时间，所以，在服务器允许的情况下，为了达到更到的性能，我们应该使用 CMS 来代替默认的 并行垃圾回收器。
  • G1垃圾回收器（G1 Garbage Collector）
    • G1 垃圾回收器是在 JDK1.7 中才正式引入的一款垃圾回收器，“科技在进步，所以一般越是先进的技术一般会更好用并且会替代陈旧的技术”，好了，玩笑归玩笑，但是 G1 的引入，目的就是为了取代 CMS 的。
    • 不要被上面 G1 的示意图误导， G1 并没有将内存进行物理划分，它只是将堆内存划分为一个个的 Region，但是也是属于分代垃圾回收器，G1 仍然会区分年轻代和老年代，并且年轻代仍然会有 Eden 区和 Survivor 区。

python内存回收机制

  • python中内存管理主要使用过 计数 + 清理 来完成的
    • 计数
      • python中对象的核心是一个结构体，内部会有一个引用计数器(ob_refcnt), 反映当前对象的名称数量，当引用计数为0的时候内存就会被释放掉。
      • 计数加一
        • 对象被创建
        • 对象被引用
        • 对象被作为参数，传入到一个函数中
        • 对象作为一个元素，存储在容器中
      • 计数减一
        • 对象别名被显示销毁 del
        • 对象别名被赋予新的对象
        • 一个对象离开他的作用域
        • 对象所在的容器被销毁或者是从容器中删除对象
    • 引用计数的优缺点
      • 优点:
        • 高效
        • 实现逻辑简单
        • 具有实时性质
      • 缺点:
        • 逻辑简单但是实现较为麻烦，每个对象需要分配单独空间来引用计数并且维护。
        • 一些场景下会较慢。当需要释放一个比较大的对象时候，比如字典，可能会耗费比较长的事件
        • 循环引用。
    • 清理
      • 1、标记清除算法
        • 算法分为“标记”和“清除”两个阶段，首先标记所有需要回收的对象，在标记完成后统一回收所有被标记的对象。有两个不足：一是效率问题，标记和清除两个过程的效率都不高；另一个是空间问题，标记清除之后会产生大量不连续的内存碎片，空间碎片太多可能会导致以后在程序运行过程中需要分配较大对象时，无法找到足够的连续内存而不得不提前触发一次垃圾收集动作
      • 2、复制算法
        • 将内存分为两块，每次只使用其中的一块。当这一块内存用完了，就将还存活的对象复制到另一块上，然后再把已使用过的内存空间一次清理掉。这样使得每次都是对一整块内存回收，内存分配时候也不用考虑内存碎片等复杂情况，只要移动堆顶指针，按顺序分配即可，实现简单，运行高效。缺点是一次只能使用一部分内存，会有一些浪费。一般新生代会选择这种算法。
      • 3、标记-整理算法
        • 复制算法存在两个问题，1）会浪费50%的空间 2）如果被使用的内存中所有对象都100%存活的极端情况，就需要有额外的空间进行分配担保，因此老年代一般不能直接选用复制算法。有人提出了另外一种“标记-整理”（Mark-Compact）算法，标记过程仍然与“标记-清除”一样，后续步骤让所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存。
      • 4、分代回收算法
        • 分代回收算法并没有什么新的思想，只是根据对象存活周期的不同将内存划分为几块。比如新生代和老年代，不同代使用不同的回收算法。比如新生代使用复制算法，而老年代使用标记-清除或标记-整理算法
  • python程序结束时会释放所有的内存么
    • 当 Python 退出时，从 Python 模块的全局命名空间引用的对象并不总是被释放。 如果存在循环引用，则可能会发生这种情况。 还有一些由 C 库分配的内存是不可能释放的(例如，像 Purify 这样的工具会抱怨这些)。然而，Python 在退出时会积极清理内存，并且确实会尝试销毁每个对象。

事物

什么是事物

  • 事物是并发控制的基本单位。是一个操作系列，这些操作要么都执行，要么都不执行，是一个不可分割的工作单位。是数据库中维护数据一致性的单位，每个事物结束之后都能保持数据的一致性。

事物的四个基本特征(ACID)

  • A: Atomicity -> 原子性
    • 事物是数据库的逻辑单位不能分割，其中的操作要么都做，要么都不做
    • 实现原理
      • undo log(回滚日志), 会执行相反操作，例如insert 就执行 delete
      • 如果事物执行失败了或调用rollback, 导事物需要回滚, 可以利用undo log之中的信息将数据回滚到修改之前的样子
  • C: Consistency -> 一致性
    • 事物执行的结果是使数据库从一个一致性状态变换到另一个一致性状态
    • 实现原理
      • 保证原子性、持久性和隔离性，如果这些特性无法保证，事务的一致性也无法保证
      • 数据库本身提供保障，例如不允许向整形列插入字符串值、字符串长度不能超过列的限制等
      • 应用层面进行保障，例如如果转账操作只扣除转账者的余额，而没有增加接收者的余额，无论数据库实现的多么完美，也无法保证状态的一致
  • I: Isolation -> 隔离性
    • 一个事物的执行不能被其他的事物干扰
    • 实现原理
      • (一个事务)写操作对(另一个事务)写操作的影响：锁机制保证隔离性
      • (一个事务)写操作对(另一个事务)读操作的影响：MVCC保证隔离性
  • D: Durability -> 持久性
    • 一个事物一旦提交，其对数据库产生的改变应该是永久性的，不能回滚
    • 实现原理
      • redo log(重做日志)
      • 由于mysql的引擎中有buffer pool机制, 其中包含了磁盘部分页的映射, 作为访问数据库的缓冲。当写入数据时候会优先写入buffer pool，定期刷入到磁盘当中，如果还没来得及刷入磁盘就宕机了，那么内存中的数据会丢失。所以每次写入buffer pool之前会有redo log首先记录更改，然后写入buffer pool, 这样宕机之后会用redo log进行内存恢复。

乐观锁和悲观锁

  • 当我们要对一个数据库中的一条数据进行修改的时候，为了避免同时被其他人修改，最好的办法就是直接对该数据进行加锁以防止并发
  • 乐观锁
    • 概念: 乐观并发控制相信事务之间的数据竞争(data race)的概率是比较小的，因此尽可能直接做下去，直到提交的时候才去锁定，所以不会产生任何锁和死锁
  • 悲观锁
    • 概念: 一种对数据的修改抱有悲观态度的并发控制方式。我们一般认为数据被并发修改的概率比较大，所以需要在修改之前先加锁。
    • 实现方式
      • 在对记录进行修改前，先尝试为该记录加上排他锁（exclusive locking）。
      • 如果加锁失败，说明该记录正在被修改，那么当前查询可能要等待或者抛出异常。具体响应方式由开发者根据实际需要决定。
      • 如果成功加锁，那么就可以对记录做修改，事务完成后就会解锁了。
      • 其间如果有其他事务对该记录做加锁的操作，都要等待当前事务解锁或直接抛出异常。
  • 如何选择乐观锁和悲观锁
    • 乐观锁并未真正的加锁，效率高，一旦粒度掌握不好，更新失败的几率就会比较高，容易发生业务失败
    • 悲观锁依赖数据库加锁，效率低，但是失败的效率也比较低
    • 随着 高并发， 高性能，高可用 的提出，悲观锁已经用的越来越少了

关系型数据库和非关系型数据库

  • 关系型数据库 RDBMS -> relational database management system
    • relational: table表
    • 概念:
      • 关系型数据库是采用关系模型来组织数据的数据库。 关系模型: 表
    • 代表:
      • mysql, sql server, postgresql, oracle etc.
    • 优点:

      • 使用方便: 使用sql进行操作
      • 易于维护: ACID降低了数据冗余和数据不一致的概率
    • 瓶颈:
      • 海量数据的读写效率

    • 应用: 一致性非常高的场景， 例如银行系统
  • 非关系型数据库 NOSQL -> not only sql

隔离级别

  • Read Uncommitted
    • 读未提交即一个事物可以读取另外一个事物未提交的数据
    • 例如: 老板给员工发工资，员工工资为1w，老板手滑点了2w，钱已经打入程序员的户口，但是事物还未提交。这时候员工去查看工资发现多了1w，老板及时发现了不对，点回滚，将数字改成了1w再提交
    • 分析: 实际工资1w，员工能看到2w(未提交前的数据)。这就是脏读
  • Read Committed
    • 一个事物要等到另外一个事物提交后才能读取数据
    • 程序员去消费，买单的时候先进行检测(程序员事物开始), 发现里面有1w, 在这个时候他妻子把钱都转出去了, 当系统开始扣费的时候再检测发现没钱了。
    • 分析: 若有事物对数据进行更新时，读操作需要等待更新事物提交之后才能读取数据, 解决了脏读取问题。但是一个事物范围内两个相同的查询返回了不同的结果，这是不可重复读
  • Repeatable Read
    • 重复读: 在开始读取数据之前，不再允许修改操作
    • 程序员消费买单时候系统检测卡里有1w, 这时候妻子不能转出金额了，就可以扣款了。
    • 分析: 重复读可以解决不可重复读问题。但是这里重复读解决的是update的问题，insert也可能出现问题, 就是幻读问题
    • 程序员开始消费，消费了2000元，妻子查看了消费记录(妻子事物开启), 发现只有2000元。这时候程序员花了10000买了个电脑，妻子打印消费记录发现是1.2w元。
  • Serializable
    • 序列化: 事物串行化顺序执行，避免了脏读，不可重复读，幻读等问题。但是效率低下，耗费数据库性能，一般不使用。

Mysql

索引

  • 什么是索引
    • 对数据库表中一个或多个列的值进行排序的结构，是帮助MYSQL高效获取数据的数据结构
    • 作用: 避免全表扫描

  • 索引的一般类型

  • 索引的数据结构
    • B-TREE
      • M阶符合的要求
        • 平衡二叉树最多两个子树，而 B 树每个节点都可以有多个子树，M 阶 B 树表示每个节点最多有M个子树。
        • 平衡二叉树每个节点只有一个数据和两个指向孩子的指针，而 B 树每个中间节点有 k-1 个关键字（可以理解为数据）和 k 个子树（ k 介于阶数 M 和 M/2 之间，M/2 向上取整）。
        • 所有叶子节点均在同一层、叶子节点除了包含关键字和关键字记录的指针外也有指向其子节点的指针，只不过其指针地址都为 null 。
    • B+_TREE
      • B+_Tree 是 B_Tree的更新(B_tree的叶子节点和非叶子节点都保留索引和数据，B+_Tree只有叶子节点保留实际数据，非叶子节点只保留索引)
    • B+_Tree有什么优点
      • 非叶子节点不会有指向记录的指针，这样一个块中可以容纳更多的索引项
          1. 可以降低树的高度
          1. 一个内部节点可以定位更多的叶子节点
      • 叶子节点之间通过指针进行连接，范围扫描非常简单
    • 为什么使用B+树，不用hash呢(hash可以一次性定位)
      • hash无法进行范围查找, 如 >= …
  • 基本的索引类型(基于mysql的InnoDB引擎)
    • 主键索引
      • 基于主键来创建B+数结构
    • 二级索引
      • 主键索引和二级索引都是维护自己的B+树结构，但是二级索引叶子节点存储的值是主键值
      • 步骤: 二级索引找到主键值 -> 主键索引找到真实数据 -> 两次B+树查找过程
    • 覆盖索引
    • 联合索引
      • 多个列进行索引
  • 索引的缺点
    • 磁盘占用
    • 维护索引对更新操作带来的耗时

drop, delete, truncate

  • 区别
    • drop一般用于删除掉表和其所有数据(元数据和数据)
    • delete一般用于删除表的部分数据(删除表中的某几条数据)
    • truncate一般用于删除表中的所有数据但是不删除表结构(删除数据，不删除元数据)

分库和分表

  • mysql的执行效率是基于服务器的处理能力，所以不能通过水平的增加服务器来进行性能的提升，所以当数据量快速增长的时候就会出现一定量的性能问题，这时候有两种解决方案
    • 提升硬件能力，比如增加存储容量，cpu等，方案成本较高，瓶颈在mysql本身
    • 分库分表，将大量的数据打散分散到不同的数据库之中，使得单一数据库的数据量变小来环节单一数据库的性能问题
  • 分库分表的类型
    • 垂直分库
      • 可以将卖家库分为商品库和店铺库，每个库存放一些信息，放于不同的服务器上，从而加速查询
      • 提升
        • 解决业务层面的耦合，业务清晰
        • 能对不同业务的数据进行分级管理、维护、监控和扩展等
        • 高并发场景下，垂直分库一定程度上提升IO、数据库连接和降低单机硬件资源的瓶颈
    • 垂直分表
      • 表中某些字段使用频率较低，例如描述类，占用空间较大，访问时间长。这时候可以考虑将一些类似这样的字段单独放入一张表之中，将访问频率高的字段放入另一张表中。
      • 提升
        • 为了避免IO争抢并减少锁表的几率，查看详情的用户与商品信息浏览互不影响。
        • 充分发挥热门数据的操作效率，商品信息的操作的高效率不会被商品描述的低效率拖累。
    • 水平分库
      • 把一个库中的数据分到多个不同的库，每个库只有这个库的部分数据，这些库可以分步在不同的服务器，从而实现负载均衡，提升性能。
    • 水平分表
      • 将一个表的数据分到同一个库的多张表中，每个表中的数据只有这个表的部分数据，能够小幅度提升性能

Redis

redis介绍

  • redis介绍
    • redis是高性能的key-value数据库
  • 与其他的key-value数据库相比有如下三个特点
    • redis支持数据的持久化，可以将内存中的数据保存在磁盘中，重启的时候可以再次加载使用
      • 全量模式 -> snapshot模式
      • 增量模式 -> AOF日志模式
    • redis不仅仅简单支持key-value啥类型数据，还支持list,set,zset,hash,string等数据结构存储
    • redis支持数据的备份, 即master-slave模式的数据备份

为什么redis非常快

  • 性能高的原因
    • 内存存储: redis使用内存存储，没有磁盘IO上的开销
    • 单线程实现: redis使用单个线程处理请求，避免了多个线程切换和锁资源争用的开销
    • 非阻塞IO: redis使用多路复用的技术，在poll, epoll和kqueue选择最优的IO实现
    • 优化的数据结构: redis有诸多可以直接应用的优化数据结构的实现，应用层可以直接使用原生的数据结果提升性能
      • redis支持的数据结构
        • 字符串: string
        • 列表: list
        • 有序集合: zset
        • 哈希表: hash
        • 集合: set

为什么redis有了持久化机制，还要用mysql呢？

  • 不是一个关系型数据库(sql对于表的复杂查询，分析还是非常有帮助的)
  • 将所有数据存储在内存中，虽然有持久化机制但是宕机之后恢复还是比较慢的

OSI模型以及TCP/IP模型

  • OIS模型: 物理层 - 数据链路层 - 网络层 - 传输层 - 会话层 - 表示层 - 应用层
  • TCP/IP模型: 网络访问层 - 网络层 - 传输层 - 应用层

DNS

  • 定义: 域名系统， 域名和IP相互映射的一个分布式数据库（集群式的）
  • 分类: 根服务器 顶级域名服务器 权限域名服务器 本地域名服务器
  • 工作流程
    • 主机向本地域名服务器的查询一般是递归查询，本地域名服务器向根域名服务器的查询一般是迭代查询
    • 递归查询: 本机向本地域名服务器发送请求，本地域名服务器去请求根域名服务器而不是本机去请求根域名服务器
      • 请求: 本机 -> 本地域名服务器 -> 根域名服务器
      • 返回: 根域名服务器 -> 本地域名服务器 -> 本机
    • 迭代查询: 本地域名服务器向根域名服务器发送请求后，根域名服务器不知道则让本机去联系顶级域名服务器
      • 请求: 本地域名服务器 -> 根域名服务器 -> 顶级域名服务器
      • 返回: 顶级域名服务器 -> 本地域名服务器

TCP与UDP

  • 区别
    • TCP面向连接, UDP无连接
    • TCP是可靠协议, UDP不可靠
    • TCP传输字节流, UDP传输数据报
    • TCP传输效率慢, UDP传输快
    • TCP首部字节20-60, UDP首部字节8
  • 应用场景
    • TCP: HTTP, FTP, SMTP
    • UDP: 包总量较小(DNS), 视频音频等多媒体及时通信, 广播等
  • TCP如何保证可靠传输
    • 校验和, 序列号, 超时重传, 流量控制, 拥塞避免等
    • 校验和: 发送端和接收端分别计算校验和，不一致则说明有问题
    • 序列号: TCP会对每一个发送的字节编号，接收方接收到数据后会对发送方发送确认应答(ACK)，并且ACK报文中会带有确认编号，告诉发送方下一次发送数据的编号
    • 超时重传: 若发送方发送数据一段时间后没有收到ACK，发送方会重新发送数据
    • 流量控制: 发送方发送数据太快会造成接收端来不及接受出现丢包。发送方会根据ACK报文中体现的接收方窗口大小来控制发送速度
    • 拥塞控制: 出现拥塞会出现丢包，如果继续重传拥塞会更加严重。慢开始 -> 拥塞避免 -> 快重传 -> 快恢复

TCP三次握手，四次挥手

  • 三次握手
    • 客户端 -> 服务端: SYN=1 seq=x 客户端进入SYN-SENT阶段
    • 服务端 -> 客户端: SYN=1 ACK=1 seq=y ack=x+1 服务端进入SYN-RCVD阶段
    • 客户端 -> 服务端: ACK=1 seq=x+1 ack=y+1 两边都进入ESTABLISHED阶段
  • 四次挥手
    • 客户端 -> 服务端: FIN=1 seq=u 客户端进入FIN-WAIT-1阶段
    • 服务端 -> 客户端: ACK=1 seq=v ack=u+1 服务端进入CLOSE-WAIT阶段 客户端进入FIN-WAIT-2阶段
    • 服务端 -> 客户端: FIN=1 ACK=1 seq=w ack=u+1 客户端进入LAST-ACK阶段
    • 客户端 -> 服务端: ACK=1 seq=u+1 ack=w+1 客户端进入TIME-WAIT阶段 服务端CLOSED
    • 客户端TIME-WAIT阶段2MSL后进入CLOSED阶段
      • 为什么是2MSL(报文最长寿命)
        • 考虑丢包问题，如果四次挥手的报文丢失了，服务端没有收到ACK会重发第三次挥手的报文，如果不等待直接关闭，报文就又丢失，会造成服务端无法正常关闭
  • TCP建立连接后，客户端故障怎么办
    • 会有TCP保活机制，发送保活探测报文，如果发送固定次数没有回应则会关闭TCP连接

TCP流量控制与拥塞控制

  • 流量控制(flow control)
    • receiver会告诉sender应该发送的最大速度
      • 比如: 一个老师在教课，一个学生在学, 学生学不动了老师还在交, 这时候会就产生问题, 所以学生应该定期给老师反馈
    • 滑动窗口机制
      • tcp会计算receiver最大能够接受的数据大小(without acknowledgement), 通过tcp头部发送给sender来调整sender的发送窗口大小
      • 三种过程
        • opening window
          • The opening window process allows more data to be in the buffer that can be sent
        • closing window
          • The closing window process indicates some sent data bytes have been acknowledged, and the sender needs not to worry about them anymore
        • shrinking window
          • The shrinking windows process denotes the decrement in the window size. It means some bytes are eligible to transmit
  • 拥塞控制(congestion control)
    • 慢开始(slow start)
      • 发送方发送数据包并逐渐增加数据包的数量，直到到达阈值。(每次增加1 MSS{max segment size}的数据包大小)
    • 拥塞避免(congestion avoidance)
      • 如果到达了阈值还是线性的增长，会导致网络拥塞，所以定义了一套到阈值之后的增长模式用来增长窗口大小
    • 拥塞检测(congestion detection)
      • 如果网络中仍存在拥塞, 需要进行检测机制
      • 两种情况
        • 估计时间内没有收到发送方发送的数据包确认
        • 接收方获得三个重复确认
  • 区别
    • 流量控制是针对sender和receiver的, 拥塞控制是针对网络堵塞的
    • 流量控制是传输层和数据链路层负责, 拥塞控制是传输层和网络层负责
    • 流量控制是为了receiver不会过载, 拥塞控制是为了网络不会过载
    • 流量控制是通过调整sender的发送速度来避免拥堵, 拥塞控制是通过传输层减少数据传输来避免拥堵

HTTP与HTTPS

  • 区别
    • 端口不同: HTTP -> 40 HTTPS -> 443
    • 安全性: HTTP无加密，安全性较差, HTTPS有加密机制，安全性较高
    • 资源消耗: HTTP较少, HTTPS较多
    • 证书: HTTP不需要, HTTPS需要
    • 协议: HTTP运行在TCP上, HTTPS运行在SSL上(SSL运行在TCP上)
  • 对称加密与非对称加密
    • 对称加密是指加密和解密使用同一密钥
      • 优点: 运算速度快
      • 缺点: 如何安全的将密钥传输给另外一方
    • 非对称加密是指加密和解密使用不同的密钥
      • 一把公钥一把私钥，公钥加密的信息需要私钥才能进行解密
      • 优点: 可以安全的将密钥传输
      • 缺点: 速度慢
  • 常见的加密算法
    • 对称性加密算法: AES、DES、3DES
    • 非对称性加密算法: RSA、DSA、ECC
  • HTTPS加密过程
    • 服务端生成一对非对称密钥，公钥发送给客户端。
    • 客户端生成对称密钥，用服务端发来的公钥进行加密，加密后发给服务端
    • 服务端使用私钥进行解密
    • 中间会有证书的作用
  • 常用的HTTP状态码
    • 1xx: 信息性状态码
    • 2xx: 成功状态码
    • 3xx: 重定向状态码
    • 4xx: 客户端错误状态码
    • 5xx: 服务器错误状态码
  • 常用的HTTP方法
    • GET: 获取资源
    • POST: 传输实体主题
    • PUT: 上传文件
    • DELETE: 删除文件
    • 可以将PUT, DELETE, POST, GET视为客户端对服务器端的增删改查
  • GET和POST的区别
    • GET用于获取资源, POST用于传输实体
    • GET的参数放在URL中, POST的参数存储在实体主体中
    • GET方法因为参数在URL中, 安全性相对较低
    • GET方法有幂等性(连续发送多次请求收到结果相同), POST方法不具备幂等性
  • 浏览器输入URL到显示主页的过程
    • 对输入的URL进行DNS解析(请求本机 -> 本地域名服务器 …), 将域名转换为ip地址
    • 和目的服务器建立TCP连接
    • 向目的服务器发送HTTP请求
    • 服务器处理请求并返回HTTP报文
    • 浏览器解析并渲染页面

Session, Cookie, Token的主要区别

  • Cookie
    • 是保存在客户端的一个小数据块，包含了用户信息。客户端向服务器发起请求的时候服务器会给客户端发送一个cookie, 当下次请求服务端时, 会携带cookie来确定身份
  • Session
    • Session是通过Cookie实现的, Session存在服务端。当客户端第一次访问服务端时，服务端会给客户端配置一个sessionid, 并将sessionid放到cookie中。当再次访问的时候会取出cookie中的sessionid, 根据sessionid获取会话中存储的信息。
  • Token
    • 客户端在浏览器第一次访问服务端时，服务端生成的一串字符串作为Token发给客户端浏览器，下次浏览器在访问服务端时携带token即可无需验证用户名和密码
  • 禁用cookie之后session还能用么
    • sessionid一般是通过cookie传递。如果cookie被禁用, 可以通过URL传递sessionid。

进程, 线程, 协程

进程与线程的区别

  • 功能上: 进程是资源调度的基本单位, 线程是程序执行的基本单位
  • 内存上: 进程拥有独立内存, 线程共享内存
  • 开销上: 进程系统开销较大, 线程较小
  • 通信上: 进程间通信较为复杂, 线程因为有共享空间, 所以通信较为简单

协程与线程的区别

  • 协程又叫微线程
  • 线程与进程都是同步机制，协程是异步机制
  • 协程是抢占式的，线程是非抢占式的。协程需要用户释放使用权切换到其他协程，同一时间只有一个协程拥有使用权，相当于单线程

进程和线程的切换流程

  • 进程
      1. 切换页表使用新的地址空间，处理器中所有已经缓存的内存地址都作废
      1. 切换内核栈和硬件上下文
  • 线程的切换第一步是不需要做的，因为线程是共享所在进程的虚拟地址空间的

进程间的通讯

  • 进程间通信: Inter Process Communication(IPC)
    • 是一种允许进程间交流和同步动作的机制
  • 一般进程有两种类型
    • 独立进程
    • 合作进程
  • 通讯方式
    • 管道:
      • 简介: 管道是一种半双工的通信方式，数据只能单向流动，而且只能在具有亲缘关系的进程间使用。通常是父子进程关系。
      • 限制: 速度慢，容量有限
      • 分类:
        • 匿名管道
          • 有固定的读端和写端
          • 只能用于具有亲缘关系进程之间的通信
        • 命名管道
          • 可以在无关进程之间交换数据
    • 共享内存(shared memory)
      • 概念:共享内存就是映射一段能被其他进程所访问的内存，这段共享内存由一个进程创建，但多个进程都可以访问。共享内存是最快的 IPC 方式，它是针对其他进程间通信方式运行效率低而专门设计的。它往往与其他通信机制，如信号两，配合使用，来实现进程间的同步和通信。
      • 优点: 能够容易控制容量，速度快
      • 限制: 需要保持同步
    • 信号量:
      • 信号量是一个计数器，可以用来控制多个进程对共享资源的访问。它常作为一种锁机制，防止某进程正在访问共享资源时，其他进程也访问该资源。因此，主要作为进程间以及同一进程内不同线程之间的同步手段。
      • 限制: 不能传递复杂消息，只能用来同步
    • 信号:
      • 信号是一种比较复杂的通信方式，用于通知接收进程某个事件已经发生。
    • 消息队列: 容量收到系统限制
      • 消息队列是由消息的链表，存放在内核中并由消息队列标识符标识。消息队列克服了信号传递信息少、管道只能承载无格式字节流以及缓冲区大小受限等缺点。
    • socket: 任何进程间都能通讯，但是速度慢
      • 套解口也是一种进程间通信机制，与其他通信机制不同的是，它可用于不同及其间的进程通信。

进程间的同步方式

  • 临界区: 通过对多线程的串行化来访问公共资源或一段代码，速度快，适合控制数据访问
  • 互斥量: 协调对一个共享资源的单独访问
  • 信号量: 控制具有有限数量用户资源
  • 事件: 用来通知线程一些事件的发生，从而启动后继任务

进程的调度策略

  • 先来先服务: 谁先来谁先调度
  • 短作业优先: 按估计运行时间最短的顺序进行调度
  • 最短剩余时间: 按照剩余时间的顺序进行调度
  • 时间片轮转: 每个进程执行一个时间片，执行完成后放到就绪队列的末尾
    • 时间长度设置过长或者过短会有什么问题呢？
      • 过长会导致对于短的请求响应时间变长
      • 过短会导致进程切换次数过多，减低cpu效率
  • 优先级调度: 为每一个进程设置优先级, 防止优先级低的进程永远等不到调度，可以随着时间的推移增加等待进程的优先级

进程状态

  • 就绪: 进程已处于准备运行的状态，即进程获得了除CPU之外的一切所需资源, 一旦得到CPU即可运行。
  • 运行: 运行状态就是进程正在CPU上运行
  • 等待: 进程正在等待某一事件而暂停运行，比如等待某资源为可用或等待I/O完成
  • 有的地方可以加上创建和终止

线程的分类

  • 内核级线程: 创建，撤销，切换都由内核实现
  • 用户级线程: 不依赖操作系统核心，由线程库来完成创建和管理

线程安全

  • 什么是线程安全
    • 指某个函数在多线程执行的环境中，能够正确的处理执行者之前的公用变量，是的程序能够正常的完成
  • 如何解决
    • 加锁呗
      1. 同步代码块
      1. 同步方法
      1. 加锁

线程共享什么，独占什么

  • 共享: 共享进程的地址空间
    • 进程代码段
    • 进程的共有数据
    • 进程打开的文件描述符
  • 独占: 每个线程有独立的堆和栈
    • 线程id
    • 寄存器组的值
    • 堆栈
    • 优先级

堆和栈

  • 栈
    • 由操作系统自动分配释放 ，用于存放函数的参数值、局部变量
  • 堆
    • 堆由开发人员分配和释放， 若开发人员不释放，程序结束时由 OS 回收，分配方式类似于链表
  • 区别
    • 管理方式不同: 栈由操作系统自动分配释放， 堆的申请和释放由程序员控制，容易内存泄漏
    • 空间大小不同: 每个进程拥有的栈大小要远远小于堆大小
    • 分配效率不同: 栈由操作系统自动分配, 会在硬件层级对栈提供支持, 分配专门的寄存器存放栈的地址，效率较高。堆是c++提供的库函数进行申请和管理，实现较为复杂，效率比较低

内核态与用户态

  • 内核态
    • 在内核模式下，执行代码可以完全且不受限制地访问底层硬件。 它可以执行任何 CPU 指令并引用任何内存地址。 内核模式通常保留给操作系统的最低级别、最受信任的功能。 内核模式下的崩溃是灾难性的； 他们将停止整个 PC。
  • 用户态
    • 在用户模式下，执行代码无法直接访问硬件或参考内存。 在用户模式下运行的代码必须委托给系统 API 来访问硬件或内存。 由于这种隔离提供的保护，用户模式下的崩溃总是可以恢复的。 在您的计算机上运行的大部分代码都将在用户模式下执行。
  • 内核态和用户态切换的开销大
    • 保留用户态现场（上下文、寄存器、用户栈等）
    • 复制用户态参数，用户栈切到内核栈，进入内核态
    • 额外的检查（因为内核代码对用户不信任）
    • 执行内核态代码
    • 复制内核态代码执行结果，回到用户态
    • 恢复用户态现场（上下文、寄存器、用户栈等）
  • 什么情况会导致用户态到内核态切换
    • 系统调用：用户态进程主动切换到内核态的方式，用户态进程通过系统调用向操作系统申请资源完成工作，例如 fork() 就是一个创建新进程的系统调用，系统调用的机制核心使用了操作系统为用户特别开放的一个中断来实现，如Linux 的 int 80h 中断，也可以称为软中断
    • 异常：当 C P U 在执行用户态的进程时，发生了一些没有预知的异常，这时当前运行进程会切换到处理此异常的内核相关进程中，也就是切换到了内核态，如缺页异常
    • 中断：当 C P U 在执行用户态的进程时，外围设备完成用户请求的操作后，会向 C P U 发出相应的中断信号，这时 C P U 会暂停执行下一条即将要执行的指令，转到与中断信号对应的处理程序去执行，也就是切换到了内核态。如硬盘读写操作完成，系统会切换到硬盘读写的中断处理程序中执行后边的操作等

死锁的概念以及产生的条件

  • 概念
    • 在两个或多个并发的进程中，每个进程持有某种资源而又等待其他进程释放它或他们保持着的资源，在没有改变这种状态之前都不能向前推进的状态
  • 产生的条件
    • 互斥: 一个资源一次只能被一个进程使用
    • 请求与保持: 一个进程因为请求资源阻塞的时候，对已经获得的资源保持不放
    • 不可剥夺: 进程获得的资源，没有使用完之前不能强行剥夺
    • 循环等待: 若干进程形成头尾相接的环形等待资源关系

  • 资源动态分配以避免死锁
  • 破除四个死锁的必要条件之一来防止死锁

分页与分段

分页

  • 分页
    • 把内存空间分成大小相同的固定的块。因为程序数据存储在不同的页面中，需要一个页表来记录映射关系，实现从页号到物理块号的映射
    • 两次访问内存: 1. 从内存访问页表 2: 根据第一次的物理地址访问内存取出数据

什么是交换空间

  • 当内存资源不足的时候，linux把某些页的内容移到硬盘上的一块空间上，释放内存空间。硬盘上的就叫做交换空间。物理内存和交换空间的总容量就是虚拟内存的可用容量
  • 用途
    • 物理内存不足的时候可以将不常用的页交换出去
    • 程序启动的时候内存页被初始化，之后不再需要

页面替换算法

  • 运行过程中要访问的页面不在内存中，需要将其调入内存中
    • 最佳算法
      • 所选择的页将是最长时间不再被访问(理想)
    • 先进先出
      • 换出的页是最先进入的页面
    • LRU
      • 将最近最久未使用的页面换出
    • 时钟算法

OSI vs TCP/IP 模型

发表于 2022-02-04 更新于 2022-02-09 分类于 计算机网络

OSI模型简介

  • 定义: OSI模型(Open Systems Interconnection Model)是用来描述网络工作原理的概念框架

第一层: 物理层(Physical layer)

  • 物理层考虑的是将底层非结构化的数据比特流(bits)以电信号的方式从发送方传输到接收方。在物理层，人们可能会发现一些"物理"资源，例如网络集线器，电缆，中继器，网络适配器等
  • 物理层用到的协议包括：蓝牙，通用串行总线等
  • 常用的设备：光纤，集线器，串口，并口

第二层: 数据链路层(DataLink layer)

  • 作为发送端: 数据链路层考虑的是将数据流转换成为信号, 通过底层硬件发送
  • 作为接收端: 数据链路层考虑的接收来自硬件的信号数据，将其集成为能够识别的帧格式，并发送给上层
  • 作用
    • 从网络层获取数据将其包装成帧，在硬件上诸位发送。从物理层获取信号并组装成帧发送给网络层
    • 提供硬件寻址机制
    • 保证数据帧的同步传输
    • 检测传输中会发生的错误，并恢复成为实际数据位
    • 流量控制使得两台机器能够相同速度交换数据
  • 数据链路层用到的协议包括: 以太网，异步传输模式，WIFI
  • 常见的设备: 网卡，交换机，桥接器

第三层: 网络层(Network layer)

  • 把传输层产生的报文段或用户数据封装成组或者包进行发送, 选择合适的网间路由和交换节点, 保证数据及时传送
  • 功能
    • 寻址: 依靠IP地址进行通信(类似数据链路层的MAC地址)
    • 路由: 在同一个网络中的内部通信并不需要网络层设备，依靠数据链路层就可以完成互相通信
  • 网络层用到的协议包括: IP, ICMP, RIP等
  • 常见的设备: 路由器, 三层交换机

第四层: 传输层(Transport layer)

  • 传输层对一个会话或连接提供可靠的传输服务
  • 功能
    • 复用: 将应用程序接受报文后封装在传输层的段中再交给网络层发送
    • 分用: 当从网络层收到数据后，必须将数据正确的传递给某个应用程序(能够对不同进程的数据加以区分)
    • 寻址: 定位应用程序
    • 流量控制/拥塞控制等
  • 传输层用到的协议包括: TCP, UDP等

第五层: 会话层(Session layer)

  • 会话负责会话检查点和恢复
  • 功能
    • 为会话实体创造连接
    • 为两个会话用户提供有组织的，同步的数据传输
    • 连接释放
  • 会话层用到的协议包括: PPTP, RPC, SOCKS等

第六层: 表示层(Presentation layer)

  • 表示层负责为不同终端的上层用于提供数据和信息正确的语法表示变换方法
  • 功能
    • 数据语法转换
    • 语法表示
    • 连接管理
    • 数据处理: 加密解密, 压缩解压, 编码解码等
  • 表示层用到的协议包括: HTTP/HTTPS, FTP/FTPS, SSH等

第七层: 应用层(Application layer)

  • 应用层直接和应用程序接口结合，并提供常见的网络应用服务
  • 应用层用到的协议包括: SMTP, DNS, POP3等

TCP/IP模型简介

  • 第一层: 网络访问层(Network Access layer)
    • 包含了OSI模型中的 数据链路层 - 物理层(二合一)
  • 第二层: 网络层(Network layer)
  • 第三层: 传输层(Transport layer)
  • 第四层: 应用层(Application layer)
    • 包括了OSI模型中的 会话层 - 表示层 - 应用层(三合一)

OSI模型 vs TCP模型

  • 不同
    • OSI七层, TCP/IP四层
    • OSI模型是概念性的模型，主要用于描述，讨论和理解网络功能。TCP/IP模型适用于解决实际问题
    • OSI模型是通用的，独立于协议的，但大多数协议都遵守它。TCP/IP模型是基于internet开发的标准协议
